# Виртуални методи и полиморфизъм. Абстрактни класове

## I. Виртуални функции

Ако имаме базов клас `Animal` и наследник `Dog`, и двата имат метод `makeSound()`.

```cpp
#include <iostream>

class Animal {
public:
    void makeSound() {
        std::cout << "Някакъв общ животински звук..." << std::endl;
    }
};

class Dog : public Animal {
public:
    // Тук дефинираме същата функция, но тя просто "скрива" базовата
    void makeSound() {
        std::cout << "Бау-бау!" << std::endl;
    }
};

int main() {
    Animal* pet = nullptr;
    
    try {
        // Указателят е от тип Animal, но сочи към обект Dog
        pet = new Dog();
        
        // Резултат: "Някакъв общ животински звук..."
        pet->makeSound();
    } catch (...) {
        // Undefined behavior
        delete pet;
        
        return 1;
    }
    
    // Undefined behavior
    delete pet;
    
    return 0;
}
```

- **Обяснение:** По подразбиране C++ използва *static binding* (ранно свързване). Компилаторът гледа типа на указателя (`Animal`), а не на обекта в паметта.
- **Решение:** Ключовата дума `virtual`. Тя казва на компилатора да провери какъв е обектът по време на изпълнение (*runtime*).

```cpp
class Animal {
public:
    // Вече е виртуална
    virtual void makeSound() {
        std::cout << "Някакъв общ животински звук..." << std::endl;
    }
    
    // Важно: виртуален деструктор
    virtual ~Animal() {}
};

class Dog : public Animal {
public:
    // Добра практика е да ползваме override
    void makeSound() override {
        std::cout << "Бау-бау!" << std::endl;
    }
};
```

- **Без `virtual`**: Решава се според типа на **указателя** (статично).
- **С `virtual`**: Решава се според типа на **обекта в паметта** (динамично/полиморфно).
- **Визуално**: Без `virtual` обектът `Dog` е "снишен" до `Animal` в очите на компилатора, докато с `virtual` той запазва своята идентичност дори зад маската на базовия клас.

## II. Как работи паметта при виртуалните функции?

- Всеки клас с виртуални методи получава скрита таблица с адреси на функции (`VTABLE`).
- Всеки обект съдържа скрит указател (`vptr`), който сочи към тази таблица.
- Програмата не скача директно към кода, а първо "пита" таблицата коя версия на функцията трябва да изпълни за конкретния обект.

Когато извикаме `pet->makeSound()`, процесорът изпълнява следните стъпки:

1. Отиди до обекта, към който сочи `pet`.
2. Вземи стойността на скрития указател (`vptr`) в началото на обекта.
3. Скочи до `VTABLE`, към която сочи този `vptr`.
4. Намери адреса на функцията `makeSound` в тази таблица.
5. Чак сега скочи към **реалния** машинен код на функцията.

## III. Полиморфизъм

Полиморфизмът е механизъм, който позволява едно и също име на метод да се свързва с различен код (имплементация) по време на изпълнение, в зависимост от конкретния клас на обекта.

### Унифицирано управление

Позволява ни да третираме група от различни обекти като едно цяло (чрез общия им базов тип), без да се интересуваме от вътрешните им детайли.

### Open/Closed Principle

Кодът ни става отворен за разширение, но затворен за промяна. Когато искаме да добавим нова функционалност, просто създаваме нов наследник. Не ни се налага да отваряме и редактираме стария, вече тестван и работещ код.

### Модулност

Системата става съвкупност от заменяеми части. Можем да подменим един модул с друг, стига те да спазват общия "договор" (базовия клас), без това да счупи останалата част от приложението.

### Премахване на условната логика

Полиморфизмът заменя безкрайните вериги от `if-else` или `switch` конструкции. Вместо да питаме: "Ти от какъв тип си?", просто извикваме метода и обектът сам "знае" как да реагира според своята природа.

> **Виртуален деструктор:** Ако имаме виртуални методи, деструкторът на базовия клас **задължително** трябва да е `virtual`. В противен случай, при `delete pet`, деструкторът на `Dog` няма да се извика и ще получим **memory leak**.

## IV. Абстрактни класове и чисти виртуални функции

Понякога базовият клас е твърде общ, за да има собствена логика. Например, какво точно е "звук на абстрактно животно"? В такива случаи използваме концепцията за "план без реализация".

### Чиста виртуална функция

Тя се дефинира с оператора `= 0` и няма тяло в базовия клас.

```cpp
// "Обещавам, че всеки наследник ще знае как да прави звук"
virtual void makeSound() = 0;
```

### Абстрактен клас

Клас, който съдържа поне една чиста виртуална функция, се нарича **абстрактен**.

- **Ограничение:** От него **не могат** да се създават обекти (`Animal a` ще даде грешка при компилация).
- **`VTABLE`:** Слотът за тази функция в таблицата на базовия клас съдържа `NULL`.
- **Цел:** Служи като строг "договор". Всеки наследник е **длъжен** или да имплементира функцията, или той самият също да бъде абстрактен.
