# Шаблони. Идея, особености, използване. Шаблони на функции

## 1. Защо се нуждаем от шаблони?

В програмирането често се срещат функции, **независими от конкретния тип данни**, върху който работят. Тези операции изпълняват **една и съща логика**, без значение дали работят с `int`, `double` или потребителски обекти.

Типични примери за такива операции са:

- Намиране на максимална/минимална стойност
- Размяна на стойности
- Сортиране или сумиране на елементи

Без шаблони сме принудени да претоварваме функции за всеки отделен тип:

```cpp
int sum(int a, int b) {
    return a + b;
}

double sum(double a, double b) {
    return a + b;
}
```

Това води до:

- Излишно дублиране на код
- Трудна поддръжка (всяка промяна в логиката трябва да се отрази на няколко места)
- По-висок риск от грешки при копиране и модифициране на кода

Шаблоните решават тези проблеми чрез механизъм за **параметризиране (обобщаване) на типовете**.

> **Основна идея:** Дефинираме логиката на функцията еднократно, като използваме абстрактен параметър, а конкретният тип данни се определя автоматично от компилатора при самото използване.

## 2. Същност на шаблоните

Шаблонът представлява **абстрактно описание** на функция или структура, в което конкретният тип данни е заменен с **параметър**. Вместо да работи с фиксиран тип (като `int`), шаблонът работи с "обещание" за тип, който ще бъде уточнен по-късно.

Общ вид:

```cpp
template <typename T>
```

- `template` → ключова дума, която указва на компилатора, че следващата дефиниция е шаблон
- `<typename T>` → дефинира списък от параметри (`T` е **символично име (placeholder)** за тип, което се използва в тялото на функцията или структурата)

> **Важно:** Шаблонът сам по себе си **не е функция** и не заема място в паметта на готовата програма. Той е **логическа схема (blueprint)**, по която компилаторът автоматично генерира истински функции при необходимост.

## 3. Шаблони на функции

### 3.1. Дефиниция

Шаблон на функция е дефиниция, при която типовете на параметрите и на връщания резултат са обобщени.

```cpp
template <typename T>
T sum(T a, T b) {
    return a + b;
}
```

Тази функция може да работи с различни типове, ако операторът `+` е дефиниран за тях.

### 3.2. Инстанциране (създаване на конкретна функция)

Процесът, при който компилаторът генерира реална функция от шаблона, се нарича **инстанциране**. Това се случва изцяло по време на компилация.

```cpp
// Създава sum<int>
sum(2, 3);

// Създава sum<double>
sum(2.5, 3.1);
```

### 3.3. Начини за извикване

#### Автоматично извеждане на типа

Компилаторът сам анализира аргументите и определя `T`.

```cpp
// T се разпознава като int
sum(10, 20);

// T се разпознава като double
sum(3.14, 2.71);
```

#### Явно задаване на типа

Програмистът ръчно указва какъв да бъде `T`. Това е полезно, когато искаме да избегнем неясноти.

```cpp
// Ще принуди компилатора да работи с int (загуба на точност)
sum<int>(2.5, 3.1);
```

### 3.4. Какво се случва при компилация?

Важно е да се разбере, че самият шаблон **не заема място** в програмата. Той е само инструкция към компилатора: *"Ако някой поиска тази логика за конкретен тип, създай му я."*

Когато компилаторът срещне `sum(2, 3)`:

- Отчита, че `2` и `3` са `int` (заключава, че `typeof(T) == int`)
- "Разгъва" шаблона и генерира истинска функция в паметта
- Ако по-късно срещне `sum(2.5, 3.1)`, той повтаря процеса и генерира **втора, напълно отделна** функция за `double`

> **Извод:** За всеки различен тип, с който извикаме шаблона, компилаторът генерира **нова версия** на функцията. Ако шаблонът не се извика никъде, той изобщо не влиза в крайния код на програмата.

## 4. Особености на шаблоните

### 4.1. Проверка при използване

Шаблонът се проверява за коректност **едва при инстанциране**.

```cpp
template <typename T>
T max(T a, T b) {
    return (a > b) ? a : b;
}
```

Ще работи за:

- `int`
- `double`

Но:

```cpp
struct A {};

// Грешка
max(A{}, A{});
```

Причината е, че няма оператор `>`.

> **Извод:** Шаблонът е "обещание", че кодът ще работи. Ако типът `T` не поддържа нужните операции, грешката се открива едва при опит за компилиране на конкретната версия.

### 4.2. Изисквания към типа

Типът `T` трябва да поддържа всички операции, използвани в шаблона:

- Аритметични операции (`+`, `-`)
- Сравнения (`>`, `<`)
- Присвояване

### 4.3. Генериране на отделни функции

Тъй като за всеки различен тип (`int`, `double`, `string`) компилаторът генерира **напълно отделна функция** в машинния код, прекомерната употреба на шаблони може да доведе до значително увеличаване на размера на изпълнимия файл (`.exe`).

### 4.4. Производителност

Тъй като цялата работа по генериране на функциите се извършва **по време на компилация**:

- Няма забавяне по време на изпълнение
- Няма допълнителна памет за логика

### 4.5. Смесване на типове

Ако имаме `sum(T a, T b)`, не можем да извикаме `sum(5, 3.14)`, защото компилаторът ще се обърка - `T` трябва да е или `int`, или `double`, но не и двете едновременно.

Решения:

- **Явно преобразуване:** `sum<double>(5, 3.14)` (принуждаваме компилатора да използва `double`)
- **Многопараметрични шаблони:** Дефинираме шаблона с два различни типа

```cpp
template <typename T, typename U>
void print(T a, U b) {
    std::cout << a << " " << b;
}
```

## 5. Защо шаблоните се пишат изцяло в хедър (`.h`) файловете?

При обикновените функции можем да сложим описанието в `.h`, а логиката в `.cpp`. При шаблоните това **не работи**.

### 5.1. Аналогия

- **Обикновена функция:** Тя е като "готово ястие" - купуваш го и го консумираш (няма значение къде е приготвено)
- **Шаблон:** Той е "рецепта"; не можеш да ядеш рецептата - трябва ти готвач (компилатор), който да прочете рецептата и да приготви ястието (кода) точно в момента, в който го поръчаш

### 5.2. Какво се случва "под капака"?

Когато компилаторът обработва файл (например `main.cpp`), той работи на стъпки:

- **Вижда извикване:** `sum<int>(2, 3)`
- **Търси рецептата:** Трябва му пълния код на шаблона, за да замени `T` с `int` и да създаде истинска функция
- **Проблемът:** Ако логиката на шаблона е скрита в някой `.cpp` файл, компилаторът **не я вижда** в този момент; той знае само, че "някъде съществува такава рецепта", но не може да я прочете, за да генерира кода
- **Резултат:** Тъй като компилаторът не е могъл да генерира кода за `sum<int>`, по-късно програмата ще даде грешка при свързване

### 5.3. Основно правило

За да може компилаторът да създаде (инстанцира) функцията за конкретен тип, той трябва да вижда **целия код на шаблона** (и заглавието, и тялото) в момента на използване.

Затова винаги пишем шаблоните изцяло в хедър файловете.

## 6. Шаблони на структури

Шаблоните на структури ни позволяват да правим **контейнери за данни**, които не се интересуват от типа на съдържанието си.

Пример:

```cpp
template <typename T>
struct Point {
    T x;
    T y;
};

// Точка с цели числа
Point<int> p1 = { 5, 10 };

// Точка с плаваща запетая
Point<double> p2 = { 3.5, 2.1 };
```

Можем да направим и структура, която просто държи някаква стойност:

```cpp
template <typename T>
struct Box {
    T content;
    
    // Функция вътре в шаблонна структура
    void print() {
        std::cout << "Съдържание: " << content << std::endl;
    }
};

Box<string> nameBox = { "John Wick" };

Box<int> ageBox = { 52 };
```

> **Важно:** За разлика от функциите, при шаблонните структури винаги оказваме типа явно.

## 7. Ключовата дума `inline`

### 7.1. Проблемът с дублирането

Тъй като шаблоните стоят в хедър файловете (`.h`), те се включват в много различни `.cpp` файлове на програмата.

Ако имаме обикновена (нешаблонна) функция в хедър файла, която не е `inline`, компилаторът ще се оплаче от грешка за дублирана дефиниция, защото тя ще се появи в абсолютно всеки файл, който я включва.

### 7.2. Магията на шаблоните

Шаблоните са автоматично `inline`.

Когато компилаторът види шаблон, той знае, че този код ще се появи на много места, и автоматично го третира като `inline`. Това му позволява да "слепи" всички копия в едно, без да дава грешка при свързване.

### 7.3. Кога пишем `inline` изрично?

Въпреки че не е задължително за самите шаблони, често се пише за яснота или при специализация:

```cpp
template <typename T>
inline T min(T a, T b) {
    return (a < b) ? a : b;
}
```

## 8. Предимства и недостатъци

| Предимства | Недостатъци |
| :--- | :--- |
| **Премахване на дублиран код:** Пишеш веднъж, ползваш за всичко | **Code Bloat:** Всяко инстанциране създава ново копие код в `.exe` файла |
| **Типова безопасност:** Компилаторът следи за типовете | **Трудни грешки:** Съобщенията за грешки при шаблони са дълги и сложни |
| **Максимална производителност:** Няма забавяне по време на работа | **Време за компилация:** Повече шаблони означават по-бавна компилация |
